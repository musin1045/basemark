# BaseMark 모바일 목표안

작성일: 2026-03-20  
기준 자료: 사용자가 제공한 `basemark_mobile_app.html`, `basemark_mobile_design_guide.md`

## 1. 결론

BaseMark의 모바일 앱은 아래 기준으로 간다.

- 모바일 우선
- 한 번에 한 가지
- 터치 친화적
- 현장 기술자 중심
- 오프라인 기본

즉, 현재의 브라우저 워크벤치는 기능 실험용으로 유지하되, 실제 현장용 셸은 사용자가 제공한 모바일 프로토타입 방향을 목표 UI로 채택한다.

## 2. 현재 웹 셸과의 차이

현재 웹 셸은 아래 성격이 강하다.

- 엔진 실험용
- 비교 흐름과 고급 편집기가 한 화면에 섞임
- 개념 설명은 좋아졌지만 실제 현장 입력 화면으로는 아직 길다
- 모바일 한 손 조작 기준으로는 과하다

사용자가 원하는 모바일 목표안은 아래 성격이다.

- 홈 화면에서 바로 작업 시작
- 항목 추가는 한 화면 순차 입력
- 검토는 별도 화면
- 프로젝트 전환과 보고서는 독립 화면
- 하단 탭 네비게이션

## 3. 목표 화면 구조

### Screen 1. 홈 / 검사 메인
- 현재 프로젝트명과 상태 배지
- 진행 상태 요약
- 빠른 액션
- 기록된 항목 목록

### Screen 2. 차이점 추가
- Step 1: 종류 선택
- Step 2: 위치 선택
- Step 3: 설명 입력
- Step 4: 검토 필요 여부

### Screen 3. 검토 전송
- 검사 요약
- 항목 목록
- 추가 노트
- 검토 전송

### Screen 4. 프로젝트 선택
- 검색
- 최근 프로젝트
- 상태 배지

### Screen 5. 보고서
- 보고서 목록
- 생성일
- 빠른 공유/다운로드

## 4. BaseMark에 맞게 꼭 남겨야 하는 것

모바일 셸이 단순해져도 BaseMark 정체성은 아래를 유지해야 한다.

- 기준점 기반 비교
- 구조 앵커 / 체크포인트 / 관측 요소 개념
- `missing`, `extra`, `position_diff`
- 후보 생성 후 사람 확정
- 비율/실측 환산 중심

즉, UI는 단순해지되 엔진 의미는 지워지면 안 된다.

## 5. 모바일 셸에서 감출 것

기본 화면에서는 아래를 감춘다.

- 원시 JSON
- 디버그용 상세 상태
- 전체 계약 입력
- 저수준 시나리오 편집

이것들은 `고급 편집` 또는 `관리자/설계 모드`로 내린다.

## 6. 모바일 셸에서 앞으로 보여줄 것

홈 화면에서 항상 보여줄 것:

- 프로젝트명
- 현재 상태 (`Draft`, `Review`, `Finalized`)
- 기록된 항목 수
- 검토 필요 항목 수
- 가장 최근 항목
- `차이점 추가`
- `검토로 보내기`

차이점 추가 화면에서 항상 보여줄 것:

- 현재 단계
- 필수 입력 여부
- 저장 가능 여부
- 취소/저장

## 7. 현재 엔진과의 연결 방식

모바일 목표안은 아래처럼 현재 엔진과 연결한다.

1. 홈 화면
- 현재 프로젝트의 요약 상태를 보여준다.

2. 차이점 추가
- 현재는 수동 입력으로 시작한다.
- 이후 단계에서 기준점/사진/실측 엔진과 연결한다.

3. 검토 전송
- 현재 후보 검토 상태와 로컬 리뷰 저장층을 재사용한다.

4. 프로젝트 선택
- 로컬 시나리오 저장소를 재사용한다.

5. 보고서
- 기존 로컬 보고서 생성 흐름을 붙인다.

## 8. 구현 원칙

- 첫 구현은 현재 웹 셸 위에 모바일 화면 레이어를 얹는 방식으로 간다.
- 기존 엔진/저장/검토 로직은 최대한 재사용한다.
- 고급 편집기는 없애지 말고 뒤로 숨긴다.
- 안드로이드 셸에서는 이 모바일 화면을 기본 홈으로 쓴다.

## 9. 다음 bounded step

다음 단계는 전체 재작성 대신 아래 범위로 제한한다.

1. 현재 셸에 모바일 하단 탭 구조 추가
2. 홈 / 항목 추가 / 검토 3화면 우선 구현
3. 현재 엔진 상태를 홈 요약 카드에 연결
4. 고급 편집기는 접기 또는 관리자 모드로 이동

이 범위면 현재 코드베이스를 버리지 않고도 사용자가 원하는 디자인 방향으로 실제 제품을 이동시킬 수 있다.
