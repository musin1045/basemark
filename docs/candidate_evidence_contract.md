# BaseMark Candidate Evidence Contract

## Purpose

This document defines the output contract for comparison candidates generated by the BaseMark engine.

BaseMark does not perform final automatic judgment.
It generates structured candidates with enough evidence for human review.

## Candidate Types

The engine supports these primary candidate types:

- `missing`
- `extra`
- `position_diff`

These are comparison results, not workflow states.

## Core Principle

The engine should emit a candidate only when it can explain the local comparison basis that produced it.

Every candidate therefore needs:

1. segment grounding
2. anchor grounding
3. normalized-coordinate grounding
4. projected evidence grounding

## Candidate Record Contract

Each candidate record should contain at least:

- `candidateId`
- `candidateType`
- `segmentId`
- `checkpointId`
- `activeAnchors`
- `normalizedBasis`
- `expectedLocation`
- `evidenceRegion`
- `reasonCode`
- `reviewHint`

### Field Meaning

- `candidateId`: stable candidate identifier
- `candidateType`: one of `missing`, `extra`, `position_diff`
- `segmentId`: local segment in which the candidate was generated
- `checkpointId`: checkpoint reference, nullable for `extra`
- `activeAnchors`: anchors used for the local comparison frame
- `normalizedBasis`: normalized coordinate model and values used for projection
- `expectedLocation`: projected target location or neighborhood
- `evidenceRegion`: image crop, region reference, or equivalent evidence pointer
- `reasonCode`: concise technical reason for the candidate
- `reviewHint`: signal for human confirmation priority

## Candidate-Specific Rules

### Missing

A `missing` candidate is valid only if:

1. a checkpoint is defined for the local segment
2. the checkpoint was projected through an accepted local alignment
3. the expected neighborhood lacks the expected element or appears empty

Suggested reason-code examples:

- `expected_region_empty`
- `expected_feature_not_found`
- `expected_structure_not_observed`

### Extra

An `extra` candidate is valid only if:

1. an observed element appears inside the active local segment
2. the observed element has no acceptable checkpoint correspondence
3. the active local frame is good enough to support that exclusion

Suggested reason-code examples:

- `unmatched_observed_element`
- `baseline_checkpoint_absent`
- `observed_structure_outside_expected_set`

### Position Diff

A `position_diff` candidate is valid only if:

1. an expected checkpoint correspondence exists
2. the observed element is present
3. the observed relative position exceeds the allowed tolerance

Suggested reason-code examples:

- `relative_offset_exceeds_tolerance`
- `height_ratio_exceeds_tolerance`
- `span_ratio_exceeds_tolerance`

## Evidence Contract

Each candidate must carry evidence that a reviewer can inspect.

Allowed evidence forms include:

- local image crop
- projected overlay region
- before/after comparison view
- structured region reference into a stored evidence image

The evidence should show both:

1. what was expected
2. what was observed

## Review Contract

The output of the comparison engine should support human review decisions such as:

- confirm
- reject
- hold
- needs_review

These review decisions are downstream workflow results, not part of the engine's raw candidate type.

## Minimum Success Condition

The engine output is acceptable for this stage if at least one candidate can be emitted with:

1. a valid local segment reference
2. a valid active anchor set
3. a normalized coordinate basis
4. a projected expected location
5. a legible evidence reference
6. a machine-readable reason code

## Non-Goals

This contract does not require:

- final autonomous acceptance or rejection
- natural-language scene explanation
- full inspection record persistence
- report formatting
