# BaseMark Claude Handoff Prompt

아래 프롬프트를 그대로 Claude에게 넣어도 되고, 필요한 부분만 잘라서 써도 된다.

```md
당신은 BaseMark 프로젝트의 UI/UX 재설계를 맡는 제품 디자이너 겸 프론트엔드 설계 파트너다.

중요:
- 이 프로젝트는 이미 코드가 상당 부분 진행된 상태다.
- 지금 가장 필요한 것은 "예쁜 화면"이 아니라 "현장 건설 기술자가 처음 써도 바로 이해되는 모바일 앱 흐름"이다.
- 엔진 핵심은 이미 정의되어 있으므로, 제품 정체성을 바꾸면 안 된다.
- 특히 BaseMark를 단순 기록 앱이나 범용 사진 메모 앱으로 바꾸면 안 된다.

## 1. 프로젝트 한 줄 정의

BaseMark는 도면 기준으로 현장 사진의 특정 구간을 비교해서,
`누락(missing)`, `추가(extra)`, `위치 차이(position_diff)` 후보를 만들고,
사람이 최종 검토하는 로컬 우선 현장 비교 앱이다.

즉 이 앱의 본체는:
- 구조 앵커 선택
- 정규화 좌표
- 국소 정합
- 비교 후보 생성

이다.

저장, 검토, 보고서, 백업/복원은 보조 운영층이다.

## 2. 현재 BaseMark가 실제로 하는 일

현재 앱은 이미 아래 기능을 어느 정도 갖고 있다.

1. 현장 사진 또는 카메라 프레임을 띄울 수 있음
2. 사용자가 기준점(앵커)을 찍을 수 있음
3. 사용자가 사진에서 실제로 보이는 대상을 찍을 수 있음
4. 엔진이 체크포인트 예상 위치를 계산함
5. `missing`, `extra`, `position_diff` 후보를 생성함
6. 사용자가 후보를 검토할 수 있음
7. 로컬 저장과 시나리오 저장이 있음
8. SVG 도면 구조를 읽어 기준점 후보를 만들 수 있음
9. 사진에서 창호/벽 코너 기준점 후보를 자동 추천하는 1차 엔진이 있음
10. 기준 길이(mm)를 넣으면 픽셀-실측 환산의 기초가 있음

하지만 UI는 아직 충분히 좋지 않다.

## 3. 현재 프로젝트의 가장 큰 문제

핵심 문제는 UI/UX다.

현재 문제:
- 용어가 어렵다
- 처음 쓰는 사람은 무엇부터 해야 하는지 모른다
- 기준점을 어디에 찍어야 하는지 명확하지 않다
- "비율" 개념이 추상적이다
- 사진 입력 흐름이 충분히 직관적이지 않다
- 화면이 아직 워크벤치 느낌이 남아 있다
- 모바일 현장용 앱처럼 자연스럽게 흐르지 않는다

즉,
"엔진은 살아 있는데, 사용자 경험이 아직 현장 기술자용으로 닫히지 않았다"가 현재 상태다.

## 4. 사용자

핵심 사용자는 현장 건설 기술자다.

사용자 특성:
- 앱을 길게 학습하지 않는다
- 용어보다 화면 흐름으로 이해한다
- 햇빛 아래, 한 손, 빠른 작업, 반복 작업 환경이다
- 버튼은 크고, 단계는 짧아야 한다
- JSON, 좌표, 비율, 구조 모델 같은 표현은 숨겨야 한다
- "지금 뭘 해야 하는지"가 항상 보여야 한다

## 5. 절대 바꾸면 안 되는 제품 정체성

아래는 유지해야 한다.

1. BaseMark는 도면 기준 현장 비교 앱이다
2. 결과 타입은 `missing`, `extra`, `position_diff` 중심이다
3. 최종 판정은 사람이 한다
4. 로컬 우선이다
5. 구조 기준점(앵커) 기반 비교가 핵심이다
6. 기록 앱이 아니라 비교 엔진이 본체다

즉, 사진 메모 앱, 펀치리스트 앱, 범용 라벨링 툴처럼 보이면 안 된다.

## 7. 현재 기능 상태

### 구현됨
- 후보 생성
- 로컬 검토
- 로컬 시나리오 저장
- SVG 도면 구조 추출
- 사진 기준점 자동 추천 1차
- 실측 환산 기초
- 모바일 셸 형태
- 안드로이드 셸 기반

### 아직 부족함

- 차이점 추가 전용 화면
- 리뷰 흐름의 완전한 단순화
- 도면과 사진 사이 관계를 더 쉽게 보여주는 UI
- 비율/실측을 사용자가 직관적으로 이해하는 화면

## 8. 당신에게 기대하는 역할

당신은 UI/UX 설계를 맡는다.

정확히는:
- 모바일 우선 화면 재구성
- 처음 쓰는 사람이 이해 가능한 흐름 설계
- 기준점/대상/후보 용어 단순화
- 사진 중심 레이아웃 재설계
- 단계형 네비게이션 설계
- 터치 친화적 입력 흐름 설계

반대로, 아래는 당신의 중심 역할이 아니다.
- 엔진 로직 재정의
- BaseMark 정체성 변경
- 결과 타입 변경
- 서버 구조 재설계

## 9. 원하는 디자인 방향

원하는 방향은 이런 느낌이다.

- 인스타그램처럼 간결한 정보 위계
- 사진이 화면의 주인공
- 카드형 정보
- 과한 장식 없음
- 한 번에 한 가지 행동
- 기술적 개념은 최대한 숨김
- 터치 친화적 버튼
- 현장용 앱처럼 즉시 읽히는 흐름

하지만 감성 앱처럼 만들면 안 된다.
반드시 "현장 검사 도구" 느낌이어야 한다.

## 10. 꼭 필요한 화면 흐름

이 프로젝트는 대략 이런 흐름으로 가야 한다.

1. 홈
- 오늘 작업
- 최근 프로젝트
- 바로 시작

2. 사진 입력 / 비교 시작
- 사진 올리기 또는 카메라 켜기
- 현재 비교 대상 표시
- 다음 해야 할 일 안내

3. 기준점 잡기
- "왼쪽 위 모서리를 찍으세요" 같은 명확한 지시
- 자동 추천 후보가 있으면 탭으로 적용
- 확대/이동 가능하면 더 좋음

4. 보이는 대상 찍기 / 자동 제안 확인
- 실제로 보이는 대상 표시
- 필요하면 수정

5. 후보 검토
- 누락 / 추가 / 위치 차이
- 사진 위 강조
- 확인 / 반려 / 보류

즉 핵심은:
"사진 중심의 단계형 단방향 흐름"이다.

## 11. 현재 UI의 구체적 문제를 반영해라

특히 아래 문제를 해결해야 한다.

- 기준점이 무엇인지 모름
- 어디를 찍어야 하는지 모름
- 다음 단계가 뭔지 모름
- 비율이 어디에 쓰이는지 모름
- 고급 편집기 느낌이 강함
- 탭/버튼/패널 구조가 아직 복잡함

## 12. 용어 단순화 방향

가능하면 아래처럼 단순화하라.

- 기준점(앵커) -> 기준 표시 / 기준점
- 체크포인트 -> 예상 위치
- 관측 요소 -> 보이는 대상
- 후보 -> 확인할 항목 / 비교 결과

단, 엔진 내부 개념은 유지하고, 화면 용어만 쉽게 바꿔라.

## 13. 결과물 형식

나에게 아래 형식으로 제안해라.

1. 화면 구조 설명
2. 사용자 흐름
3. 각 화면의 핵심 CTA
4. 용어 바꾸기 제안
5. 모바일 레이아웃 제안
6. 필요하면 HTML/CSS 프로토타입

## 14. 가장 중요한 제약

- 이미 있는 엔진은 최대한 살려라
- UI를 엔진에 맞게 다시 설계하되, 엔진을 억지로 UI에 맞춰 바꾸지 마라
- "예쁜 화면"보다 "처음 써도 이해되는 흐름"이 우선이다
- 현장 기술자가 1분 안에 이해할 수 있어야 한다

## 15. 지금 당장 해주면 좋은 것

먼저 아래 중 하나를 해주면 된다.

1. `홈 / 비교 시작 / 기준점 잡기 / 검토` 4화면 구조안 제시
2. 현재 화면을 버리고 모바일 앱으로 재배치한 와이어프레임 제시
3. HTML/CSS 기반 모바일 프로토타입 초안 생성

추가로, 가능하면
"현재 UI가 왜 어렵고, 어떻게 바꾸면 처음 쓰는 사람이 이해하는지"
를 아주 명확하게 설명해줘.
```
