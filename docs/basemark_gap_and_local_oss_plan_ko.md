# BaseMark 갭 분석 및 로컬 오픈소스 도입안

## 1. 지금 설계에 들어가 있지만, 실제로는 아직 제대로 적용되지 않은 것

### 1-1. 구조 기준점 선택은 아직 "역할 고정"이 아니라 "점 찍기"에 가깝다
- 현재 UI에는 `창호 좌상단`, `벽 좌측 모서리` 같은 프리셋이 들어갔지만, 엔진은 여전히 좌표 점 두 개를 받아 단순 투영하는 수준이다.
- 즉 "같은 구조선의 짝"인지, "바깥 프레임 vs 안쪽 창짝"을 구분했는지, "왜 이 점이 구조 기준점인지"를 시스템이 강하게 보장하지 못한다.
- 실제 의미 있는 앵커 계약은 아직 미완성이다.

### 1-2. 도면 기준 입력이 실제 도면 편집 흐름으로 닫혀 있지 않다
- 체크포인트는 `spanRatio`, `heightRatio`로 들어가 있지만, 이것을 실제 도면 위에서 정의하는 작성 도구가 없다.
- 현재는 JSON/폼 값으로 직접 넣거나 예시 시나리오에 의존한다.
- 설계상 "도면 기반"인데, 실제 구현은 아직 "수동 수치 입력 기반"에 가깝다.

### 1-3. 국소 정합이 너무 단순하다
- 현재 엔진은 두 기준점만으로 `two_anchor_span_projection`을 수행한다.
- 이 방식은 첫 프로토타입으로는 맞지만, 원근 왜곡, 회전, 기울기, 렌즈 차이, 촬영 위치 차이를 충분히 흡수하지 못한다.
- 실제 현장에서는 기준점 두 개만으로는 흔들릴 가능성이 크다.

### 1-4. 관측 요소 추출이 아직 전부 수동이다
- 사진에서 보이는 스위치 박스나 설비 박스를 자동 후보화하지 못한다.
- 사용자가 직접 관측 요소를 찍어야 하므로, 현재 BaseMark는 "엔진 프로토타입"이지 "현장 반자동 비교 시스템"은 아니다.

### 1-5. 후보 생성 근거가 아직 너무 얇다
- 지금은 `elementKind`와 거리 기반 반경 비교가 핵심이다.
- 실제 사용 단계에서는 "왜 missing인가", "왜 extra인가", "왜 position_diff인가"를 더 풍부한 시각 근거로 보여줘야 한다.
- 예: 구조선과의 상대 거리, 후보 주변 크롭, 관측 요소 스냅샷, 기준점 쌍 기준 오차량.

### 1-6. 실시간 카메라 오버레이는 있지만 안정화가 덜 됐다
- 카메라 연동은 들어가 있지만, 프레임 간 추적, 흔들림 보정, 터치 오차 보정, 실폰 해상도 차이에 대한 안정화는 부족하다.
- 지금은 "카메라가 붙은 프로토타입"이지 "현장용 AR형 오버레이"는 아니다.

### 1-7. 리뷰 흐름은 있으나, 현장 사용 언어는 아직 덜 다듬어졌다
- `confirm / reject / hold` 흐름은 있지만, 실제 현장 언어로는 `확인 / 반려 / 보류 / 재촬영 필요` 같은 더 직관적인 분기가 필요할 수 있다.
- 또한 아직 일부 화면은 고급 편집기 관점이 남아 있다.

## 2. 실제 사용한다면 생길 문제

### 2-1. 앵커를 잘못 찍으면 전체 결과가 흔들린다
- 가장 큰 문제다.
- 예를 들어 창호 기준이라면서 한쪽은 바깥 프레임, 다른 쪽은 안쪽 창짝을 잡으면 비율 자체가 잘못된다.
- 이 경우 엔진이 틀린 게 아니라 입력 기준이 무너진다.

### 2-2. 비전문 사용자는 "어디를 찍어야 하는지"에서 막힌다
- 지금도 사용자가 바로 물었던 문제가 이것이다.
- 기준점은 단순 클릭이 아니라 "역할 있는 구조점 선택"인데, 이 차이를 UI가 더 강하게 가르쳐야 한다.

### 2-3. 사진 조건이 바뀌면 단순 투영이 쉽게 틀어진다
- 가까이 찍은 사진, 비스듬한 사진, 광각 렌즈 사진은 오차가 커진다.
- 기준점이 두 개뿐이면 작은 오차가 체크포인트 전체에 증폭될 수 있다.

### 2-4. 관측 요소 수동 입력은 피로도가 높다
- 현장에서는 반복 촬영과 반복 클릭이 생긴다.
- 실제 도입 단계에서는 최소한 "관측 요소 후보 추천"까지는 들어가야 체감 가치가 올라간다.

### 2-5. 결과 근거가 약하면 신뢰를 잃는다
- 현장 사용자는 "왜 이게 missing으로 떴는지"를 봐야 한다.
- 현재처럼 타입과 좌표만으로는 충분하지 않을 수 있다.

### 2-6. 도면 작성 도구가 없으면 프로젝트 확장이 막힌다
- 지금은 시나리오 1개를 사람이 만들 수는 있지만, 프로젝트 여러 구간을 빠르게 정의하기 어렵다.
- 결국 도면 기반 체크포인트 작성 UI가 필요하다.

## 3. 현재 나온 시각 분석 시스템들이 주로 쓰는 방식 중, BaseMark에 로컬로 가져올 수 있는 것

### 3-1. 고전 기하 기반 정합
- 가장 먼저 가져와야 한다.
- 기준점 2개 클릭에만 의존하지 않고, 직선/모서리/코너를 보조 근거로 사용해 정합을 안정화할 수 있다.
- 후보:
  - OpenCV `LineSegmentDetector` / `FastLineDetector`
  - OpenCV `findHomography`, `estimateAffinePartial2D`
- 장점:
  - 로컬 CPU에서도 충분히 돈다.
  - 모바일/노트북 환경에 부담이 적다.
  - BaseMark의 "구조 관계 기반 비교"와 잘 맞는다.
- 권장 용도:
  - 창호 프레임, 벽선, 문틀선 검출
  - 기준점 주변 선분 보조
  - 2점 정합을 4점/선분 기반 정합으로 확장

### 3-2. 로컬 특징점 매칭
- 사진과 기준 이미지 사이의 대응점을 더 안정적으로 찾는 데 쓸 수 있다.
- 후보:
  - LightGlue
  - LoFTR
- 장점:
  - 사람 클릭을 줄이고, 대응점 후보를 자동 추천할 수 있다.
  - 국소 구간 정합 품질을 높이기 좋다.
- 주의:
  - LoFTR는 정확도는 좋지만 가볍다고 보긴 어렵다.
  - LightGlue 쪽이 현장 로컬 앱에는 더 현실적이다.

### 3-3. 가벼운 객체 검출
- 관측 요소를 전부 손으로 찍게 하지 말고, 스위치 박스/노출 박스/콘센트 박스 같은 대상은 먼저 후보를 뽑아주는 데 적합하다.
- 후보:
  - Ultralytics YOLO
- 장점:
  - 로컬 GPU가 있으면 충분히 빠르다.
  - 작은 커스텀 데이터셋으로도 특정 대상만 잡는 데 유리하다.
- 권장 용도:
  - 관측 요소 후보 자동 제안
  - extra 후보의 초벌 탐색

### 3-4. 클릭 기반 분할
- 사용자가 "여기가 창호 프레임" 또는 "이 박스가 대상"이라고 한 번 누르면, 경계를 더 정확히 따주는 보조 도구다.
- 후보:
  - MobileSAM
- 장점:
  - BaseMark의 사람-확정 흐름과 궁합이 좋다.
  - SAM2보다 훨씬 로컬 친화적이다.
- 권장 용도:
  - 창호 프레임과 안쪽 창짝 구분
  - 관측 요소의 경계 잡기

### 3-5. OCR은 보조로만 사용
- 텍스트가 있는 설비 라벨이나 박스 표기 확인용으로는 유용하다.
- 후보:
  - PaddleOCR
- 장점:
  - 로컬 실행성이 좋고, 범용 OCR 성숙도가 높다.
- 주의:
  - BaseMark의 본체를 OCR에 의존하면 안 된다.
  - 구조 비교의 본질은 선/점/상대 위치다.

### 3-6. 로컬 주석/검수 UI 참고용
- 엔진 자체보다 "사람이 어떻게 빠르게 보고 수정하는가"를 가져오기 좋은 오픈소스다.
- 후보:
  - X-AnyLabeling
  - CVAT
  - Label Studio
- 권장 판단:
  - 통째로 제품 본체로 쓰는 건 무겁다.
  - 하지만 클릭-기반 주석 UX, 후보 검수 UX, 단축 흐름은 참고 가치가 높다.

## 4. BaseMark에 실제로 가져오면 좋은 조합

### 1순위
- OpenCV 기반 선/코너 보조 정합
- 이유:
  - 가장 가볍고
  - BaseMark 핵심인 구조 기준 정합과 직접 연결되고
  - 안드로이드/노트북 로컬 실행에 부담이 적다.

### 2순위
- YOLO 기반 관측 요소 후보 제안
- 이유:
  - 수동 클릭 피로를 줄이는 효과가 바로 크다.

### 3순위
- LightGlue 기반 국소 대응점 추천
- 이유:
  - 기준점 추천과 장면 정합 품질 개선에 도움이 된다.

### 4순위
- MobileSAM 기반 경계 분할
- 이유:
  - 창호 프레임/박스 영역을 더 정확히 잡을 수 있다.
  - 다만 V1 필수는 아니다.

### 보조
- PaddleOCR
- 이유:
  - 라벨/텍스트 보조 확인에는 좋지만, 코어 엔진 우선순위는 아니다.

## 5. 권장 구현 순서

1. 기준점 선택을 "자유 클릭"에서 "역할 있는 짝 선택"으로 강화
2. OpenCV 선/코너 기반 보조 정합 추가
3. 체크포인트 비율을 도면에서 시각적으로 정의하는 도구 추가
4. YOLO로 관측 요소 후보 자동 제안
5. LightGlue로 대응점 추천
6. MobileSAM은 필요 시 도입

## 6. 결론

- 지금 BaseMark는 "정체성과 최소 엔진이 있는 프로토타입"이다.
- 아직 설계의 핵심 중 실제로 덜 닫힌 부분은:
  - 구조 기준점 의미 고정
  - 도면 기반 체크포인트 작성
  - 강건한 국소 정합
  - 반자동 관측 요소 추출
- 실제 현장에 바로 쓰면 가장 먼저 부딪히는 문제는 "어디를 기준점으로 찍어야 하는가"와 "사진 조건이 달라질 때 오차를 어떻게 줄일 것인가"다.
- 그래서 BaseMark에 가장 먼저 가져와야 할 로컬 오픈소스 방식은, 무거운 범용 AI가 아니라:
  - OpenCV 기반 구조선/기하 정합
  - 가벼운 객체 검출
  - 필요 시 특징점 매칭
  순서다.
